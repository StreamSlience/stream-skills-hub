# command-assistant 优化分析与发展方向

## 一、当前实现原理

### 核心架构

```
触发层（SKILL.md frontmatter）
    ↓ 关键词匹配（指令名 / 表名 / 路径）
加载层（按需读取）
    ├── index.md → docs/<cmd>.md
    ├── tables-index.md → tables.md[offset:limit]
    └── snippets/<cmd>.md
    ↓
感知层（对话内被动识别）
    ↓ 用户确认
暂存区（staging/）
    ↓ 用户确认
正式目录（docs/ snippets/ context/）
    ↓
索引维护（index.md + *-index.md）
```

### 设计亮点

1. **按需加载**：通过行范围索引（`*-index.md`）避免全文加载大文件，节省 token
2. **暂存区隔离**：所有更新先进 `staging/`，用户确认后才归档，防止误写
3. **通用与业务分离**：`docs/` 存通用说明，`snippets/` 存业务片段，职责清晰
4. **双触发机制**：被动感知（对话内）+ 钩子触发（后台），覆盖不同使用场景

## 二、当前可优化的问题

### 2.1 extract-to-staging.sh 提取质量低

**问题**：脚本用 `grep -A1 '```bash...'` 提取代码块，只取代码块后一行，无法完整提取多行命令，且无法区分"用户输入的命令"和"Claude 生成的示例命令"。

**影响**：钩子触发机制（B 机制）实际上几乎无法产出有价值的内容，`pending.txt` 里的内容质量很差，导致 `hook_enabled` 默认为 `false`，这个机制形同虚设。

**优化方向**：
- 改为让 Claude 在对话结束时（Stop hook）直接生成结构化的 JSON 摘要写入 staging，而不是用 bash 脚本做文本解析
- 或者完全依赖被动感知机制，移除低质量的 bash 提取脚本

### 2.2 commit-staging.sh 合并策略粗糙

**问题**：合并时直接 `cat src >> dst`，追加到文件末尾，没有去重逻辑。如果同一个 snippet 被多次暂存，会产生重复内容。

**优化方向**：合并前检查目标文件中是否已存在相同的 `## 场景名` 标题，有则跳过或提示用户选择覆盖/跳过/合并。

### 2.3 行范围索引维护是手动的

**问题**：SKILL.md 要求"每次 context/*.md 变更后重新扫描计算行号"，但这依赖 Claude 在每次更新后手动执行，容易遗漏，导致索引与实际文件不同步。

**优化方向**：在 `commit-staging.sh` 中加入自动重建索引的逻辑，合并完成后自动扫描并更新 `*-index.md`，不依赖 Claude 记得执行。

### 2.4 触发描述过于宽泛，误触发率高

**问题**：frontmatter description 中列举了 `awk, psql, free, clickhouse-client` 等具体工具名，但实际上任何提到命令名的对话都可能触发，包括用户只是在讨论概念而非需要查询知识库的场景。

**优化方向**：在 SKILL.md 中增加"不触发"条件说明，例如：用户只是在解释概念、不需要执行命令时不触发；或者增加一个轻量的"是否需要查知识库"判断步骤。

### 2.5 context/ 类型固定，扩展性差

**问题**：当前 context/ 只支持 tables / paths / services 三种类型，每种类型都需要手动创建对应的 `.md` 和 `-index.md` 文件，新增类型成本高。

**优化方向**：将 context 类型做成可配置的，在 `config.yaml` 中声明支持的类型，由初始化脚本自动创建对应文件结构。

### 2.6 没有搜索能力

**问题**：当前只支持精确匹配（指令名 / 表名），如果用户说"查一下上次那个查慢查询的命令"，无法通过关键词模糊定位到对应 snippet。

**优化方向**：在 index.md 中为每个 snippet 增加 tags 字段，支持按标签模糊检索。

## 三、未来发展方向

### 方向 A：知识库质量提升

**目标**：让积累的内容更有价值，而不只是堆砌命令。

- **自动生成使用说明**：当用户首次使用某个指令时，Claude 自动从官方文档或自身知识生成 `docs/<cmd>.md`，不需要用户手动触发
- **snippet 关联场景**：每个 snippet 记录"在什么问题下用到了这条命令"，而不只是命令本身，形成"问题 → 命令"的映射
- **版本标注**：记录命令适用的工具版本（如 `psql 14+`），避免在不同环境下踩坑

### 方向 B：多项目 / 多环境隔离

**目标**：支持不同项目使用不同的业务上下文。

- `snippets/<project>/` 按项目隔离业务片段
- `context/<env>/` 按环境（dev/staging/prod）隔离路径和服务映射
- 在 `config.yaml` 中支持 `active_project` 配置，切换当前激活的项目上下文

### 方向 C：知识库同步与共享

**目标**：让团队成员可以共享通用知识，同时保留个人业务片段。

- `root_dir` 指向 git 仓库，支持 `git pull/push` 同步
- 区分"个人私有"和"团队共享"两个层级：
  - 共享层：`docs/`（通用指令说明）+ `context/`（团队共用的表结构）
  - 私有层：`snippets/`（个人业务操作习惯）
- 提供 `sync` 命令：拉取团队共享层，推送个人贡献

### 方向 D：主动推荐

**目标**：从被动查询变为主动辅助。

- 统计每个 snippet 的使用频率，在相关场景下主动推荐高频命令
- 当用户描述问题时（"我要查最近一小时的风险事件"），主动匹配并展示相关 snippet，不需要用户记得去查
- 识别用户重复输入相同命令的模式，提示"这条命令已出现 3 次，是否保存为 snippet？"

### 方向 E：与其他技能联动

**目标**：让 command-assistant 成为其他技能的上下文提供者。

- `das-rest-skill` 需要查接口路径时，从 `context/paths.md` 自动注入
- `systematic-debugging` 排查问题时，自动加载相关服务的 snippet
- 提供统一的"上下文注入"接口：其他技能声明需要哪类上下文（tables/paths/services），command-assistant 负责按需加载

## 四、优先级建议

| 优化项 | 优先级 | 理由 |
|---|---|---|
| 修复 commit-staging.sh 去重逻辑 | 高 | 直接影响数据质量，改动小 |
| commit-staging.sh 自动重建索引 | 高 | 消除手动维护索引的遗漏风险 |
| 移除或重写 extract-to-staging.sh | 中 | 当前实现质量低，但 hook_enabled 默认关闭，影响有限 |
| 增加 snippet tags + 模糊检索 | 中 | 提升可用性，但需要改造现有数据结构 |
| 多项目隔离（方向 B） | 中 | 对有多个项目的用户价值高 |
| 主动推荐（方向 D） | 低 | 依赖足够的积累量，早期数据少时价值有限 |
| 团队共享（方向 C） | 低 | 需要 git 仓库配合，个人使用场景不需要 |
